# ARPEX Detector Module Design

## File

```text
arpex/detector.py
```

## Purpose

The detector is the core component of ARPEX.

Its responsibility is to monitor ARP traffic on the network, identify suspicious behavior, verify potential attacks, and generate standardized security events.

The detector does NOT:

* Store data in the database
* Write logs
* Display alerts
* Render dashboard content
* Perform UI operations

Its only responsibility is network monitoring and event generation.

---

# High Level Flow

```text
Network Interface
        ↓
ARP Packet Received
        ↓
Extract IP and MAC
        ↓
Known Device?
        ↓
No ----------------→ NEW_DEVICE Event
        ↓
Yes
        ↓
MAC Changed?
        ↓
No ----------------→ Ignore
        ↓
Yes
        ↓
Verify via ARP Request
        ↓
Confirmed?
        ↓
No ----------------→ Ignore
        ↓
Yes
        ↓
Generate ARP_SPOOFING Event
```

---

# Detection Scope

ARPEX will detect spoofing against:

* Routers
* Laptops
* Phones
* Printers
* IoT Devices
* Any network device

The detector is not limited to gateway spoofing.

---

# Event Types

## NEW_DEVICE

Severity:

```text
LOW
```

Generated when a previously unseen device appears on the network.

Example:

```text
192.168.1.25
AA:BB:CC:DD:EE:FF
```

---

## ARP_SPOOFING

Severity:

```text
HIGH
```

Generated when a device is confirmed to be impersonating another device.

---

## GATEWAY_SPOOFING

Severity:

```text
CRITICAL
```

Generated when the victim is identified as the default gateway.

---

# Verification Process

ARPEX must not immediately trust a suspicious packet.

When a MAC mismatch is detected:

```text
Known:
192.168.1.10 → AA:AA:AA

Observed:
192.168.1.10 → BB:BB:BB
```

ARPEX sends:

```text
Who has 192.168.1.10?
```

If the response confirms the original mapping:

```text
192.168.1.10 is at AA:AA:AA
```

the spoofing alert is generated.

This verification step reduces false positives.

---

# Alert Cooldown

Duplicate alerts should be suppressed.

Cooldown:

```text
10 seconds
```

Example:

```text
12:00:01 Attack detected
12:00:03 Attack detected
12:00:05 Attack detected
```

Only the first alert is generated.

After 10 seconds:

```text
12:00:15 Attack detected
```

A new alert may be generated.

---

# Device Learning

When a new device appears:

```text
192.168.1.50
AA:BB:CC:DD:EE:FF
```

ARPEX:

1. Generates a NEW_DEVICE event
2. Stores device information
3. Waits for user approval during baseline creation

---

# Baseline Policy

First run:

```text
No automatic trust.
```

User must approve discovered devices before they become trusted.

---

# Vendor Identification

The detector will support offline MAC vendor identification.

Example:

```text
00:50:56
↓
VMware
```

No online API calls are required.

Vendor data is obtained from a local OUI database.

---

# Evidence Generation

When spoofing is confirmed:

ARPEX should trigger evidence collection.

Example:

```text
captures/attack_001.pcap
```

Only suspicious traffic is stored.

Normal traffic is not continuously recorded.

---

# Attacker Tracking

The detector should maintain attacker information.

Example:

```text
MAC: BB:BB:BB:BB:BB:BB

First Seen:
2026-06-05

Last Seen:
2026-06-05

Attack Count:
12
```

This information will later be stored in the database and displayed in the dashboard.

---

# Standard Event Structure

Every event generated by detector.py should follow:

```json
{
  "timestamp": "",
  "event_type": "",
  "severity": "",

  "victim_ip": "",

  "expected_mac": "",
  "observed_mac": "",

  "vendor": "",

  "evidence_file": "",

  "interface": ""
}
```

All other modules will consume this format.
